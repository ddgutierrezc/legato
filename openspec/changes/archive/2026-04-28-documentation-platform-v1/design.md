# Design: documentation-platform-v1

## Technical Approach

Create a dedicated Starlight app at `apps/docs-site/` and treat it as the canonical public docs surface. Public guidance is migrated into docs-site pages; root/package READMEs become thin summaries; `docs/maintainers/*` and `docs/releases/*` stay canonical for maintainer operations; `docs/architecture/spikes/*` stays archive-only.

## Architecture Decisions

| Decision | Options | Choice | Rationale |
|---|---|---|---|
| Public app location | `apps/docs-site/`, reuse root `docs/`, README-only | `apps/docs-site/` | Matches monorepo app placement, avoids collision with mixed-content `docs/` and existing `docs/node_modules/`, and isolates Starlight dependencies from release/runbook files. |
| Public IA | Flat markdown set, mirror repo tree, audience-first sections | Audience-first sections | Consumers navigate by job-to-be-done, not repo topology. Top-level sections: `getting-started/`, `packages/contract/`, `packages/capacitor/`, `reference/`, `releases/`. |
| Sidebar strategy | Fully autogenerated, fully manual, hybrid | Hybrid manual groups + directory index pages | Stable top-level navigation with low drift; section-local files can grow without reshaping global IA every time. |
| Canonical authority | README-first, docs-site-first, root `docs/` mixed | docs-site-first for Public | Satisfies spec authority order: docs-site is canonical for Public; READMEs summarize and link; maintainer/release docs remain canonical only for internal audiences. |

## Data Flow

`repo source docs` → `classification matrix` → `docs-site public pages`
`README/package README` → `summary + canonical links`
`docs/maintainers`, `docs/releases`, `docs/architecture/spikes` → remain in root, excluded from public nav

Public release surface flows from `CHANGELOG.md` into `apps/docs-site/src/content/docs/releases/index.mdx`; release evidence JSON/ops records remain root-only.

## Interfaces / Contracts

- Public content contract: every docs-site page maps to one audience (`Public`) and one canonical source set.
- Maintainer/archive exclusion contract: no file under `docs/maintainers/`, `docs/releases/notes/`, `docs/releases/contracts/`, `docs/releases/*evidence*`, or `docs/architecture/spikes/` may be imported into `apps/docs-site/src/content/docs/**`.
- README contract: root and package READMEs keep orientation, install snippets, and canonical links only; deep tutorials belong in docs-site.

## Testing Strategy

| Layer | What to Test | Approach |
|---|---|---|
| Unit | Sidebar/content classification config | Node tests over allowed/blocked path lists. |
| Integration | README ↔ docs-site drift | Extend current docs drift style tests to assert canonical links and “thin README” rules. |
| Integration | Internal leakage prevention | Scan docs-site content and config for blocked root-doc paths/patterns (`releases/notes/*.json`, evidence/runbook files, spikes). |
| E2E | Public IA presence | Verify mandatory sections and public release surface exist in built content tree/config. |

## Migration / Rollout

Batch 1: scaffold `apps/docs-site/` and nav only.  
Batch 2: migrate public onboarding from `README.md`, `packages/contract/README.md`, `packages/capacitor/README.md`.  
Batch 3: add reference/release surfaces from `CHANGELOG.md`; keep `docs/releases/*` untouched.  
Batch 4: convert READMEs to thin entrypoints.  
Batch 5: add drift/leakage verification.

Archive-only retainers: `docs/architecture/spikes/*`, release evidence records (`docs/releases/notes/*.json`, go/no-go records, gap matrices, deferral registers, closure evidence). Maintainer representation remains root-repo docs plus contributor-facing links, not public sidebar sections.
